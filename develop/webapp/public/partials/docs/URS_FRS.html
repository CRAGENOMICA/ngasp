User And Functional Requirements Specification

Introduction

This document combines two specifications: the User Requirements Specification (URS) and the Functional Requirements Specification (FRS).

The objective of the User Requirements Specification document is to define the formal requirements and quality attributes that the system is expected to meet. The requirements defined in this document will provide the details necessary to furnish an effective basis for the acceptance criteria for testing.

The objective of the Functional Requirements Specification is to document the operations and activities that a system must be able to perform. It includes:

  •  * Descriptions of data to be entered into the system
  •  * Descriptions of operations performed by each screen
  •  * Descriptions of work-flows performed by the system
  •  * Descriptions of system reports or other outputs
  •  * Who can enter the data into the system
  •  * How the system meets applicable regulatory requirements

The Functional Requirements Specification is designed to be read by a general audience. Readers should understand the system, but no particular technical knowledge should be required to understand the document.

Purpose

The purpose of this document is to define the user requirements. It is a way that assures that both the requesters and the developers understand the other's requirements. It also functions as a blueprint for completing a project with as little cost growth as possible. This is often referred to as the "parent" document because all subsequent project management documents, such as design specifications, statements of work, software architecture specifications, testing and validation plans, and documentation plans, are related to it.
